# Flow 21 — Pool operator monitor view

After flow 20, the operator wants a single read-only snapshot of how
their bond is performing on pox-5: live sats currently committed, share
of the bond, current cycle's accrued (but unclaimed) rewards, and the
rewards-per-token snapshot at the last settlement. Member-level
accounting lives in the pool contract; this flow only covers the pox-5
surface.

"Total sats currently committed to this bond" is the **live** effective
total — `fetchTotalSharesStakedForCycle` keyed by bond, which tracks
exits and unstakes. The cumulative-launch snapshot
(`fetchTotalSbtcStakedForBond`, frozen at D0) is **not** what an
operator dashboard wants; it's the right read for the pre-D0 view in
flow 17. `signer-rewards-per-token-settled-for-cycle` is genuinely a
snapshot — it only advances when `claim-rewards` settles a cycle.

The rewards model is `get-earned -> uint` (plus separate
`get-signer-rewards-per-token-settled-for-cycle` / `get-signer-unclaimed-
rewards-for-cycle` reads). There is no per-signer "paid" counter — once
`claim-rewards` runs it zeroes the unclaimed entry and bumps the settled
rpt. Rewards-paid lifetime is therefore an indexer concern, not a
contract read.

```ts
import {
  currentDistributionCycle,
  fetchEarnedByBond,
  fetchPoxInfo,
  fetchSignerRewardsPerTokenSettledByBond,
  fetchSignerSharesStakedByBond,
  fetchTotalSharesStakedForCycle,
} from '@stacks/bitcoin-staking';

const network = 'mainnet';
const signerManager = 'SP000…COMMUNITY-POOL.v1';
const bondIndex = 7;

const pox = await fetchPoxInfo({ network });
const [bondLiveTotal, mySharesBond, rptSettled, earned] = await Promise.all([
  fetchTotalSharesStakedForCycle({ index: bondIndex, isBond: true, network }),
  fetchSignerSharesStakedByBond({ signerManager, bondIndex, network }),
  fetchSignerRewardsPerTokenSettledByBond({ signerManager, bondIndex, network }),
  fetchEarnedByBond({ signerManager, bondIndex, network }),
]);
const distCycle = currentDistributionCycle(pox);

const dashboard = {
  cycle: pox.rewardCycleId,
  distributionCycle: distCycle,
  bondTotalSbtc: bondLiveTotal,
  mySbtcInBond: mySharesBond,
  myShareBps: bondLiveTotal === 0n ? 0 : Number((mySharesBond * 10_000n) / bondLiveTotal),
  earnedPending: earned,         // bigint — accrued + pending, settled at next `claim-rewards`
  rewardsPerTokenSettled: rptSettled, // latest settled rpt snapshot
};
```

The `earnedPending` value is what the pool's `claim-rewards` (flow 7)
will pull on the next call. `myShareBps` is informational — pool-internal
member weights drive the actual onward distribution.

Per-staker share reads let an operator drill into a single member's
contribution to the pool's bond stake:

```ts
import { fetchStakerSharesStakedByBond } from '@stacks/bitcoin-staking';

const memberShare = await fetchStakerSharesStakedByBond({
  staker: poolContract,        // the on-chain staker IS the pool
  signer: signerManager,        // pool's signer-manager
  bondIndex,
  network,
});
```

For pools the staker is always the pool contract itself, so this read
returns the pool's full sats — member splits live off-pox-5. The same
read is more directly useful in flow 8 (individual stakers).

> FLAG: lifetime "rewards paid" per signer/bond is no longer a contract
> read — the on-chain state only carries the *unclaimed* counter plus the
> settled rpt. Operator dashboards that need a cumulative-paid figure
> must derive it from indexed `claim-rewards` print events.

## Lifecycle variants

Read-only dashboard — runs at any time, but the *interesting* fields differ per the bond's D-day. A pool typically holds positions in multiple staggered bonds at once, so the operator UI usually renders **one row per active bond**, each at its own D-day.

| D-day (per bond row) | What's meaningful on the dashboard |
| --- | --- |
| D-30 → D-7 | `bondTotalSbtc = 0`, `mySbtcInBond = 0`. Only `fetchBondAllowance` (flow 4) is useful here — pre-registration view. |
| D-7 → D0 | `mySbtcInBond` populated after the pool's `register-for-bond`; `bondTotalSbtc` still climbing as other partners register. `myShareBps` is provisional. |
| D0 → D172 | Steady-state. All fields meaningful: `earnedPending` advances each distribution cycle (~weekly); `rewardsPerTokenSettled` snapshots the latest settled rpt. `bondTotalSbtc` is now frozen against new registrations but may decrement if any party early-exits. The canonical operator view. |
| Andon pause window (250 burn blocks post-distribution) | `earnedPending` may be non-zero but `claim-rewards` (flow 7) is gated — surface a "payout paused" badge rather than promising imminent claim. |
| D172 → D182 (re-lock window) | Distributions continue; pool can call `unstake-sbtc` for partial sBTC withdrawal. The withdrawal triggers `settle-rewards` so `earnedPending` is crystallized into the unclaimed counter before shares decrement. |
| D183+ | Historical row. `earnedPending` should be drained via final `claim-rewards`; `mySbtcInBond` reads as the closed position. |

`fetchStakerSharesStakedByBond` always returns the pool's own aggregate — member splits never appear in pox-5 reads regardless of D-day.

# Flow 19 — sBTC pool member deposits

In PoX-5 the community tranche is **sBTC-only via a pool** (Launch Scope
§3). Native-BTC bonds are individual-only (Launch Scope §2). At the
contract level, `register-for-bond` allows one membership per principal
— so the **pool contract is the principal**, not the members. Members
deposit STX + sBTC into the pool's contract; the pool calls
`register-for-bond` itself (flow 20).

This flow is therefore **off-pox-5**: it talks to the pool contract,
not to `pox-5`. The exact shape is pool-defined; the SDK only provides
the read-only pieces a member needs to inspect the pool first.

```ts
import { fetchBondAllowance } from '@stacks/bitcoin-staking';
import { fetchCallReadOnlyFunction } from '@stacks/transactions';

const network = 'mainnet';
const poolContract = 'SP000…COMMUNITY-POOL.v1';

// 1. Sanity-check that the pool itself is allowlisted on the bond.
const poolAllowance = await fetchBondAllowance({
  bondIndex: pool.bondIndex,
  address: poolContract,
  network,
});
if (poolAllowance === 0n) throw new Error('pool not allowlisted');

// 2. Pool-defined deposit. The pool contract pulls STX + sBTC from the
//    member, issues a position receipt, and tracks pro-rata weight.
//    Shape varies by operator — example uses a hypothetical
//    `(deposit (amount-ustx uint) (amount-sats uint))`.
await callPoolFunction({
  contract: poolContract,
  function: 'deposit',
  args: [memberUstx, memberSbtcSats],
  sender: member,
  network,
});

// 3. The pool batches members and calls pox-5 once before D0 (flow 20).
//    sBTC is locked in the pool contract for the full bond period and is
//    not composable during the bond.
```

Member-side reads after the pool has registered:

```ts
// Members track their position via the pool's read-only fns, not pox-5.
const myPosition = await fetchCallReadOnlyFunction({
  contractAddress: poolContract.split('.')[0],
  contractName: poolContract.split('.')[1],
  functionName: 'get-position',
  functionArgs: [/* member principal */],
  senderAddress: member.stxAddress,
  network,
});
// → pool-defined: { sharesUstx, sharesSats, claimableSbtc, … }
```

Reward distribution is the pool's responsibility — pox-5 pays the pool's
signer (flow 7), and the pool splits onward to members.

## Lifecycle variants

Anchored to the **pool's active bond D-days**. Member deposits land in the pool contract, not pox-5 — but the pool can only register once, so when members may meaningfully deposit is dictated by the bond timeline. Exact deposit-window enforcement is **pool-defined** (off-pox-5); the table below reflects typical pool behavior.

| D-day (the pool's bond) | Behavior |
| --- | --- |
| D-30 → D-7 | Pool allowlisted, not yet registered. Pool contracts typically open the member deposit window here. `fetchBondAllowance` returns a non-zero cap. |
| D-7 → D0 | Final deposit window — must finish before the pool calls `register-for-bond` (flow 20). Late deposits risk missing the pool's batch. |
| D0 → D172 | Deposits typically closed at the pool level — pool is locked into the bond at its aggregated total. Member-side reads (`get-position`) remain valid. |
| D172 → D182 (re-lock window) | sBTC is still locked in the pool contract. Pool-defined whether withdrawals open; `unstake-sbtc` is available to the pool contract at any time but is a pool-operator decision (flow 20). |
| D183+ | Bond closed. Member redemptions are pool-defined. |

A pool participating in multiple staggered bonds will see this table evaluated **per bond** — a member may face an "open deposit window" for bond N+1 while their existing position is mid-bond on N.

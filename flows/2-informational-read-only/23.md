# Flow 23 — Bond calendar / schedule helper

Convert between bond index, reward cycle, and burn height. Useful for
notification timing, "X days until next bond opens," "in re-lock
window," and indexer pagination. The cycle/burn-height conversions are
pure math over `PoxInfo` (no contract round-trip); only the bond
existence check requires a fetch.

```ts
import {
  bondPeriodToBurnHeight,
  bondPeriodToRewardCycle,
  fetchPoxInfo,
  isBondActiveAtHeight,
  isInPreparePhase,
  rewardCycleToBurnHeight,
} from '@stacks/bitcoin-staking';

const network = 'mainnet';
const bondIndex = 7;

const pox = await fetchPoxInfo({ network });
// Bond-math helpers derive `firstBondPeriodCycle` from `pox.contractVersions[]`
// internally and throw if pox-5 has not yet activated on-chain.

const openBurnHt = bondPeriodToBurnHeight({ bondIndex, poxInfo: pox });
const firstRewardCycle = bondPeriodToRewardCycle({ bondIndex, poxInfo: pox });
const closeBurnHt = rewardCycleToBurnHeight({
  cycle: firstRewardCycle + 12, // BOND_LENGTH_CYCLES
  poxInfo: pox,
});

const reLockWindowBlocks = Math.floor(pox.rewardCycleLength / 2); // mainnet 1,050 (≈ 7.3d)

const now = pox.currentBurnchainBlockHeight;
const inPrepare = isInPreparePhase({ burnHeight: now, poxInfo: pox });
const isActiveNow = isBondActiveAtHeight({
  bondIndex,
  burnHeight: now,
  poxInfo: pox,
});

const schedule = {
  bondIndex,
  status:
    now < openBurnHt
      ? 'announced'
      : now < closeBurnHt - reLockWindowBlocks
        ? 'active'
        : now < closeBurnHt
          ? 're-lock-window'
          : 'closed',
  blocksUntilOpen: Math.max(0, openBurnHt - now),
  blocksUntilClose: Math.max(0, closeBurnHt - now),
  inPreparePhase: inPrepare,
  isActiveNow,
  openBurnHeight: openBurnHt,
  closeBurnHeight: closeBurnHt,
};
```

The re-lock window is canonically `rewardCycleLength / 2` (mainnet
1,050 blocks ≈ 7.3 days at the end of every bond, BTC unlocks before
STX). `inPrepare` is the gate flow 3's `unstake` and most update flows
hit.

`firstBondPeriodCycle` is derived internally by every bond-math helper from
`pox.contractVersions[]` (via `firstPox5RewardCycle`) — per `pox-5.clar:312-314`
the contract's `first-pox-5-reward-cycle` data-var is also the first bond
period cycle, and the same value is already carried on `/v2/pox` in
`contract_versions[].first_reward_cycle_id` for the `pox-5` row (no extra
round-trip). Helpers throw if pox-5 has not yet activated on-chain.

## Lifecycle variants

This flow exists to **map heights to a status label** — lifecycle is
the whole point. Inputs (`openBurnHt`, `closeBurnHt`, `now`) never
change for a given bond; what moves is `now`. All variants below come
from the same fetch — only the derived `status` and the
`blocksUntilOpen` / `blocksUntilClose` deltas differ.

| `now` relative to bond | Derived status | Notable derived fields |
| --- | --- | --- |
| Pre-D-30 | (bond doesn't exist — `bondPeriodToBurnHeight` still returns a height, but the bond hasn't been declared on-chain; pair with `fetchBond` for a real existence check) | `blocksUntilOpen` very large; `isActiveNow: false`. |
| D-30 → D0 (`now < openBurnHt`) | `announced` | `blocksUntilOpen > 0`, `blocksUntilClose > 0`, `isActiveNow: false`. |
| D0 → D172 (`now ≥ openBurnHt` and `now < closeBurnHt - reLockWindowBlocks`) | `active` | `blocksUntilOpen: 0`, `isActiveNow: true`. |
| D172 → D182 (`now` within `reLockWindowBlocks` of `closeBurnHt`) | `re-lock-window` | `blocksUntilClose` ≤ ~1,050 (mainnet). BTC is unlocked on L1 here; `l1UnlockHeight` from `fetchBondL1UnlockHeight` should be ≤ `now`. STX still locked. |
| D183+ (`now ≥ closeBurnHt`) | `closed` | `blocksUntilClose: 0`, `isActiveNow: false`. Bond record stays readable; this flow only emits the label. |

`inPreparePhase` is orthogonal to bond D-days — it cycles every reward
cycle (~14 days) regardless of where any bond sits in its arc. Same for
`isInPreparePhase`-gated update flows (3, etc.).

The re-lock boundary uses `pox.rewardCycleLength / 2` per the
read-only docs (mainnet 1,050 blocks ≈ 7.3 days, testnet 525).

## API design proposal

Today: 2 parallel fetches. Tracks back to `specs/pox-5-api-endpoint-design-review.md` §4 (`/v2/pox` already exposes pox-version metadata).

| Current call | Replaced by |
| --- | --- |
| `fetchPoxInfo` (node `/v2/pox`) | unchanged — single node call |
| `fetchFirstPox5RewardCycle` | **eliminated**. `firstBondPeriodCycle` is `pox.contractVersions.find(v => v.contractId.endsWith('.pox-5')).firstRewardCycleId` — already on `/v2/pox` once pox-5 activates (verified pre-pox-5 against `api.mainnet.hiro.so/v2/pox` — the pattern is `pox-4` today with `first_reward_cycle_id: 84`). |

Net: **2 → 1 call.**

Implemented: `fetchFirstPox5RewardCycle` has been removed in favor of the pure helper `firstPox5RewardCycle(poxInfo)`, which derives the value from `pox.contractVersions[…]` already returned by `/v2/pox`.

## Render the phase ranges

The status-label ladder above collapses the bond timeline to a single
label for "now." UIs that want to render the whole arc (progress bar,
phase chips, countdown timeline) can ask for the ranges directly via
the **unstable / UI-experimental** `bondPhaseRanges` helper — same
inputs, same math, just laid out as a list. Boundaries marked
*convention* in its JSDoc (`announced` start, `open` start) are
display-only; only D0 and the L1 CLTV expiry are contract-enforced.

```ts
import { bondPhaseRanges } from '@stacks/bitcoin-staking';

const phases = bondPhaseRanges({ bondIndex, poxInfo: pox });
// [
//   { name: 'announced',      startBurnHeight, length, endBurnHeight },
//   { name: 'open',           ... },
//   { name: 'active',         ... },
//   { name: 're-lock-window', ... },
//   { name: 'closed',         ... }, // capped at BOND_LENGTH_CYCLES * rewardCycleLength
// ]

// Drive a progress-bar UI: total span + per-phase widths.
const timelineStart = phases[0].startBurnHeight;
const timelineEnd = phases[phases.length - 1].endBurnHeight;
const totalBlocks = timelineEnd - timelineStart;
const segments = phases.map(p => ({
  name: p.name,
  offsetPct: ((p.startBurnHeight - timelineStart) / totalBlocks) * 100,
  widthPct: (p.length / totalBlocks) * 100,
  isCurrent: now >= p.startBurnHeight && now < p.endBurnHeight,
}));
```

`endBurnHeight` is exclusive (`startBurnHeight + length`), so adjacent
phases share a boundary cleanly. The `closed` phase is capped to one
bond length of blocks for finite rendering; bonds remain readable past
that cap.

# Flow 8 — Read a user's current position

No transactions. A wallet or dashboard fetches everything needed to render
"what does this address have locked?" in one shot.

```ts
import {
  fetchAccountStatus,
  fetchBondMembership,
  fetchPoxInfo,
  fetchStakerInfo,
} from '@stacks/bitcoin-staking';

const network = 'mainnet';
const address = user.stxAddress;

const [pox, account, stxOnly, bond] = await Promise.all([
  fetchPoxInfo({ network }),
  fetchAccountStatus({ address, network }),
  fetchStakerInfo({ address, network }), // STX-only position (if any)
  fetchBondMembership({ address, network }), // paired-bond position (if any)
]);

const position = {
  burnHeight: pox.currentBurnchainBlockHeight,
  cycle: pox.rewardCycleId,
  liquidUstx: account.balance - account.locked,
  lockedUstx: account.locked,
  unlockHeight: account.unlockHeight,
  stxOnly: stxOnly.staked ? stxOnly.details : null,
  bond: bond ?? null,
};
```

A new contract read `get-staker-shares-staked-for-cycle(staker, is-bond, index, signer)`
exposes the per-staker share contributed to a given signer in a given cycle —
useful when a user is delegated across multiple signers and the dashboard
wants a per-signer breakdown rather than just the wallet-level locked total:

```ts
import { fetchStakerSharesStakedForCycle } from '@stacks/bitcoin-staking';

const myShareWithSigner = await fetchStakerSharesStakedForCycle({
  staker: address,
  signer: signerManager,
  rewardCycle: pox.rewardCycleId,
  network,
});
```

## Lifecycle variants

Phases here are anchored to whatever bond the user is registered in (if
any). A wallet with no position is phase-independent — empty across all
of network time. A wallet with a paired-bond position threads through
the bond's D-day arc.

| Phase | What this flow returns |
| --- | --- |
| No position ever | `account.locked == 0`, `stxOnly: null`, `bond: null`. STX-only and paired branches are both null; the only signal is `liquidUstx`. |
| Allowlisted, not yet registered (D-7 → D0 for the user's bond) | Still `stxOnly: null`, `bond: null` — the membership row only materialises on `register-for-bond`. The allowlist row lives on a separate map (flow 17/19 territory), not surfaced here. |
| Active paired bond (D0 → D172) | `bond` populated with `{bondIndex, amountUstx, signer, isL1Lock, amountSats}`. `account.locked` reflects the STX side of the pair; `amountSats` carries the BTC shares directly off the membership row (no extra read needed). |
| Re-lock window for the user's bond (D172 → D182) | Same shape as active. `isL1Lock: true` positions are BTC-unlocked on L1 but the membership row is unchanged on L2; the L1-liveness signal (`still_locked_l1`) is **not** in this flow — needs the indexer-derived field on the §3.1 endpoint (flow 9 polls mempool.space). |
| Bond closed (D183+) | Membership map row is cleared on full unstake / sBTC-unstake / L1 early-exit settle. Expect `bond: null` again. Lifetime "I was in bond N" is event-history only — not a chain-tip read. STX-only leg (if any) is unaffected. |
| STX-only staker (any phase) | `stxOnly` carries `{amountUstx, firstRewardCycle, numCycles, signer}`; orthogonal to bond D-days. Goes to `null` once `firstRewardCycle + numCycles` is past. |

Cross-phase: this flow only sees chain-tip state. "What did the user
*do*?" (closed bonds, past STX-only cycles, expired caller grants)
needs an event-history endpoint outside the read-only surface.

# Flow 17 — Browse an upcoming bond

A partner or pool operator inspects the next bond's parameters before
enrolling: capacity, target APY, ratio, open burn height, and their
own allowlist entry.

```ts
import {
  bondPeriodToBurnHeight,
  fetchBond,
  fetchBondAllowance,
  fetchPoxInfo,
  fetchTotalSbtcStakedForBond,
} from '@stacks/bitcoin-staking';

const network = 'mainnet';
const address = user.stxAddress;
const bondIndex = 7;

const [pox, bond, filledSbtc, allowance] = await Promise.all([
  fetchPoxInfo({ network }),
  fetchBond({ bondIndex, network }),
  fetchTotalSbtcStakedForBond({ bondIndex, network }),
  fetchBondAllowance({ bondIndex, address, network }),
]);

const openBurnHeight = bondPeriodToBurnHeight({
  bondIndex,
  poxInfo: pox,
});

const summary = {
  bondIndex,
  status: openBurnHeight > pox.currentBurnchainBlockHeight ? 'announced' : 'open',
  openBurnHeight,
  blocksUntilOpen: Math.max(0, openBurnHeight - pox.currentBurnchainBlockHeight),
  capacitySats: bond?.capacitySats,
  filledSbtc,
  remainingSbtc:
    bond?.capacitySats !== undefined ? bond.capacitySats - filledSbtc : undefined,
  targetRateBps: bond?.targetRateBps,
  stxValueRatio: bond?.stxValueRatio,
  minUstxRatioBps: bond?.minUstxRatioBps,
  myAllocationSats: allowance,
};
```

## Lifecycle variants

This is the canonical "different at every D-day" flow — the four
fetches stay the same, but what's populated and what the caller should
render moves substantially through the bond's life.

| Phase | What this flow returns |
| --- | --- |
| Pre-D-30 (bond doesn't exist) | `fetchBond` returns `undefined`; `fetchTotalSbtcStakedForBond` returns `0n`; `fetchBondAllowance` returns `0n`. Render 404 / "not yet announced". Don't compute `remainingSats` against an undefined capacity. |
| D-30 → D-7 (announced) | Bond record present (`targetRateBps`, ratio fields, `earlyUnlockBytes`). Allowlist seeded — `allowance` non-zero for allowlisted callers, `0n` otherwise. `filledSbtc == 0n`, `blocksUntilOpen > 0`. Status: `announced`. Capacity = sum of allowlist entries (derived server-side; SDK leaves `capacitySats?` optional). |
| D-7 → D0 (registration window open, filling) | Same record fields. `filledSbtc` rises as partners register; `blocksUntilOpen` shrinks to 0. Allowlisted-but-not-yet-registered callers still see their `allowance`. Status: `open`. |
| D0 → D172 (active) | Steady state. `filledSbtc` at or near capacity. The flow says nothing about distributions / earned rewards — those are flow 7/21 territory. Allowlist still readable but mostly historical. |
| D172 → D182 (re-lock window, ~7.3d on mainnet) | Chain-tip reads are unchanged on L2: bond record, total staked, allowance all identical to active phase. Need `fetchBondL1UnlockHeight` (flow 23) to know BTC has unlocked on L1; this flow alone can't tell you you're in the re-lock window. |
| D183+ (closed, historical) | Bond record persists in `protocol-bonds` (map entries aren't cleared). `fetchTotalSbtcStakedForBond` stays at its D0 closing snapshot — `protocol-bonds-total-staked` is only written by `register-for-bond`, so exits and unstakes after D0 don't move it. For currently-effective shares post-exit, switch to `fetchTotalSharesStakedForCycle({ rewardCycle, bondIndex })`. Allowlist still readable. Status flips to `closed` only via height comparison — this flow doesn't surface it; derive via flow 23. |

`fetchTotalSbtcStakedForBond` reads `protocol-bonds-total-staked`, whose
only write site is `register-for-bond` — each call sets the entry to
`current(total-shares-staked-for-cycle for this bond) + new sats`, a
snapshot refreshed on every registration. During the D-7 → D0 window the
snapshot can rebase off a lower value if `unstake-sbtc` /
`announce-l1-early-exit` land between registrations. After D0,
`ERR_BOND_ALREADY_STARTED` blocks further registrations and the value is
frozen at the last registration's snapshot — exits and unstakes after D0
don't move it. For **currently-effective shares** post-exits or unstakes,
switch to `fetchTotalSharesStakedForCycle({ rewardCycle, bondIndex })`.
Fixture B7 stores the snapshot; a parallel "effective shares" fixture
should be added for the post-exit case.

## API design proposal

Today: 4 parallel fetches. Tracks back to `specs/pox-5-api-endpoint-design-review.md` §2 (`/bonds/:index`) and §4 (node vs API split).

| Current call | Replaced by |
| --- | --- |
| `fetchPoxInfo` (node `/v2/pox`) | unchanged — single node call. `firstBondPeriodCycle` is auto-derived inside the bond-math helpers from `pox.contractVersions[]`. |
| `fetchBond` | absorbed by `GET /extended/v3/staking/pox-5/bonds/:index?principal=:caller` |
| `fetchTotalSbtcStakedForBond` | embedded as `total_sbtc_registered` field on the same response |
| `fetchBondAllowance` | embedded as `my_allowance: { max_sats } \| null` on the same response (only when `?principal=` is set) |

Net: **4 → 2 calls** (1 node + 1 API).

The bond endpoint stays principal-indexed in spirit: `?principal=` is optional context. Without it the response omits `my_allowance`; with it, the endpoint serves the caller's allowlist slot inline. Same path, same cache key when principal-less.

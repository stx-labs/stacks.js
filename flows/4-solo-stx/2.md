# Flow 2 — Update an STX-only stake

The same user extends their lock by 4 cycles, adds 10,000 STX, and rotates
to a new signer-manager — all in one call. Each dimension is opt-in: omit
(or pass `0` / `0n`) to skip. To rotate without changing cycles/amount,
pass the new `signerManager` and the staker's current `oldSignerManager`
(re-read via `fetchStakerInfo` first).

```ts
import { buildStakeUpdate, fetchStakerInfo } from '@stacks/bitcoin-staking';
import { broadcastTransaction, fetchNonce, signTransaction } from '@stacks/transactions';

const network = 'mainnet';
const info = await fetchStakerInfo({ address: user.stxAddress, network });
if (!info.staked) throw new Error('not currently staked');

const tx = buildStakeUpdate({
  signerManager: 'SP000…SIGNER.signer-manager-v2', // new (or pass current to keep)
  oldSignerManager: info.details.signer,           // must match recorded signer
  cyclesToExtend: 4,
  amountIncrease: 10_000_000_000n,
  fee: 10_000n,
  publicKey: user.stxPublicKey,
  nonce: await fetchNonce({ address: user.stxAddress, network }),
  network,
});

const signed = signTransaction(tx, user.stxPrivateKey);
await broadcastTransaction({ transaction: signed, network });
```

> FLAG: SDK currently types `cyclesToExtend?: number` / `amountIncrease?: IntegerType`
> as optional with `0` fallback. There is an open SDK note in `build.ts` about
> whether a pure-rotate call (both zeros on an already-extended position) clears the
> `check-pox-lock-period` guard — no client-side gate exists today.

## Lifecycle variants

Reward-cycle-relative, not bond-D-day-relative. `stake-update` is a builder op (extend, top up, rotate signer); like other builder ops it's gated on the prepare phase.

| Phase | Behavior |
| --- | --- |
| Active reward cycle | Canonical window for all three dimensions. Extend / top-up / rotate take effect from `currentRewardCycle + 1` onward. |
| Prepare phase (last ~100 burn blocks) | Signer-manager rotation and any change that mutates next-cycle signer weight is rejected — `is-in-prepare-phase` blocks the signer-set transition. Pure top-ups to an already-active position behave like the active-cycle case but still race the prepare boundary; gate the UI conservatively. |
| `num_cycles` already at max (96) | `cyclesToExtend > 0` reverts — extend is a no-op once the indefinite-stake cap is reached. |
| `oldSignerManager` mismatch with `staker-info.signer` | `ERR_INVALID_OLD_SIGNER_MANAGER (u36)` — always re-read `fetchStakerInfo` before composing the tx. |

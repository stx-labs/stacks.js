# Flow 1 — Solo STX-only stake

A user locks STX for a single reward cycle by delegating to a signer-manager
contract. No BTC, no pairing — authorization is handled by the chosen
signer-manager's `validate-stake!` (no per-tx SIP-018 signature on `stake`).

```ts
import { buildStake, fetchPoxInfo } from '@stacks/bitcoin-staking';
import { broadcastTransaction, fetchNonce, signTransaction } from '@stacks/transactions';

const network = 'mainnet';
const pox = await fetchPoxInfo({ network });

const tx = buildStake({
  signerManager: 'SP000…SIGNER.signer-manager', // chosen signer-manager contract
  amountUstx: 50_000_000_000n,           // 50,000 STX (signer-set minimum)
  numCycles: 1,
  startBurnHt: pox.currentBurnchainBlockHeight,
  publicKey: user.stxPublicKey,
  fee: 10_000n,
  nonce: await fetchNonce({ address: user.stxAddress, network }),
  network,
});

const signed = signTransaction(tx, user.stxPrivateKey);
await broadcastTransaction({ transaction: signed, network });
```

## Lifecycle variants

Solo-STX is tied to **reward-cycle phases**, not bond D-days — STX-only stakes are not bond-anchored. The relevant timing axis is the signer cycle and its 100-block prepare-phase tail.

| Phase | Behavior |
| --- | --- |
| Active reward cycle | Canonical case. `stake` queues the lock for `first-reward-cycle = currentRewardCycle + 1`; counts toward signer weight starting next cycle. |
| Prepare phase (last ~100 burn blocks of a cycle) | Signer-set transition window. New stakes targeting the *next* cycle are at the deadline — submit before this window to avoid `is-in-prepare-phase` gates in `validate-stake!`. |
| Below `SIGNER_SET_MIN_USTX` | Below 50,000 STX lands in `signer-pending-staked-ustx-per-cycle` ("in queue / not yet active") until topped up via flow 2. |
| Andon pause window (250 burn blocks post-distribution) | No effect on staking itself — only gates downstream `claim-rewards`. |

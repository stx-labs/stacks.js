# Flow 3 — Unstake (STX-only)

The user sets their position to unlock at the end of the current cycle.
The contract reverts with `ERR_UNSTAKE_IN_PREPARE_PHASE` if called during
the prepare phase, so gate on `isInPreparePhase` first.

`unstake` takes a `signer-manager-trait` arg — it must match the staker's
currently recorded signer (`ERR_INVALID_OLD_SIGNER_MANAGER`). Re-read
`fetchStakerInfo` to compose it.

```ts
import {
  buildUnstake,
  fetchPoxInfo,
  fetchStakerInfo,
  isInPreparePhase,
} from '@stacks/bitcoin-staking';
import { broadcastTransaction, fetchNonce, signTransaction } from '@stacks/transactions';

const network = 'mainnet';
const [pox, info] = await Promise.all([
  fetchPoxInfo({ network }),
  fetchStakerInfo({ address: user.stxAddress, network }),
]);
if (!info.staked) throw new Error('not currently staked');

const inPrepare = isInPreparePhase({
  burnHeight: pox.currentBurnchainBlockHeight,
  poxInfo: pox,
});
if (inPrepare) throw new Error('cannot unstake during prepare phase');

const tx = buildUnstake({
  oldSignerManager: info.details.signer,
  publicKey: user.stxPublicKey,
  fee: 10_000n,
  nonce: await fetchNonce({ address: user.stxAddress, network }),
  network,
});

const signed = signTransaction(tx, user.stxPrivateKey);
await broadcastTransaction({ transaction: signed, network });
```

## Lifecycle variants

Reward-cycle-relative, not bond-D-day-relative. The prepare-phase guard is the only hard timing gate; everything else is cycle-boundary semantics.

| Phase | Behavior |
| --- | --- |
| Active reward cycle (outside prepare phase) | Canonical case. Marks the position to unlock at the next cycle boundary; STX stays locked through the current cycle and contributes signer weight until then. |
| Prepare phase (last ~100 burn blocks) | Contract reverts with `ERR_UNSTAKE_IN_PREPARE_PHASE`. The explicit `isInPreparePhase` guard above mirrors this — fail fast in the SDK rather than burning a fee on a revert. |
| Already unstaking | `num-cycles` has already been frozen by a prior `unstake` so the position points at `current-cycle + 1`; a second call is idempotent in spirit but the contract still runs `remove-staker-from-cycles` and rewrites `staker-info`. Re-read `fetchStakerInfo` first to avoid burning fees on no-op rewrites. |
| Position queued (below `SIGNER_SET_MIN_USTX`, only in `signer-pending-staked-ustx-per-cycle`) | `unstake` is callable. `add-staker-to-signer-cycles` writes `staker-signer-cycle-memberships` unconditionally even for queued stakers, so `remove-staker-from-cycles` finds the membership and unwinds the pending entry. |

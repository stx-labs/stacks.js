# Flow 15 — Andon Cord (pause a scheduled payout)

> **⚠ Out of scope for `@stacks/bitcoin-staking`.** The pause/payout-window
> surface lives upstream of the pox-5 contract (3-of-5 ops multisig,
> off-chain orchestration) — neither `buildPausePayout` nor
> `fetchPayoutWindow` exists in `pox-5.clar` today, and they are **not
> planned** for this SDK. This flow is preserved as a sketch for
> reference only.

Between the end of a distribution cycle and the height at which
`calculate-rewards` can fire, ops have a buffer window (enforced
upstream — the contract itself has no fixed guard) to halt a payout
if anomalies surface. Pause cannot redirect — only halt. Restoring a
paused payout may require a hard fork. Authorization is a 3-of-5 ops
multisig.

Both the read and the pause builder are sketches — neither is in
`pox-5.clar` today.

```ts
import {
  buildPausePayout, // [MISSING surface — sketch]
  fetchPayoutWindow, // [MISSING surface — sketch]
} from '@stacks/bitcoin-staking';
import { broadcastTransaction, fetchNonce, signTransaction } from '@stacks/transactions';

const network = 'mainnet';

// End users / ops dashboards: render "queued / confirmed / paused" state.
const window = await fetchPayoutWindow({ network });
// → { distCycle, scheduledHeight, blocksRemaining, canPause, paused }

if (!window.canPause) return; // window closed; payout already fired

const tx = buildPausePayout({
  distributionCycle: window.distCycle,
  reason: 'anomaly: T1 coverage ratio below threshold',
  publicKey: ops.stxPublicKey, // signed by 3-of-5 ops multisig
  fee: 10_000n,
  nonce: await fetchNonce({ address: ops.stxAddress, network }),
  network,
});
await broadcastTransaction({ transaction: signTransaction(tx, ops.stxPrivateKey), network });
```

## Lifecycle variants

Bond-agnostic — the andon cord gates a **distribution cycle**, not a specific bond. Pausing halts the cycle's whole waterfall across every bond active at that `calculation-height` simultaneously.

The flow is cycle-clock-driven, not bond-D-day-driven. Each distribution cycle (~weekly, twice per reward cycle) opens its own pause window between cycle end and the height at which `calculate-rewards` can fire. From `rewards.mdx`: the contract itself has no fixed +250 guard; the buffer is enforced upstream and is the operational andon window.

| Phase (per distribution cycle) | What's different |
| --- | --- |
| Pre-window | `fetchPayoutWindow` returns no live window for the next cycle yet. Nothing to pause. |
| Window open (`calculation-height` not yet reached) | `canPause: true`. Ops multisig (3-of-5) may call `buildPausePayout`. Across the bond universe this is the *only* moment to halt the cycle. |
| Window closed by `calculate-rewards` landing | `canPause: false`. Payout has crystallised into `signer-unclaimed-rewards-for-cycle`; flow 7 claims will succeed. Pausing this cycle is no longer possible (would require a hard fork to undo). |
| Window closed by pause tx | Cycle's `calculate-rewards` reverts going forward; signer legs for that cycle remain at `0` indefinitely (or until a hard fork restores them). Subsequent distribution cycles are unaffected — the next window opens normally. |

Unclear from current docs — the exact `buildPausePayout` and `fetchPayoutWindow` surfaces, the precise window length, and whether the contract will eventually carry the gate natively. The flow file itself notes both surfaces are sketches.

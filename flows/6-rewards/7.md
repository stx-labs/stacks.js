# Flow 7 — Claim weekly rewards (signer side)

Distributions run twice per cycle (~weekly). First someone calls
`calculate-rewards` to settle the cycle's waterfall; then the signer pulls
its accumulated sBTC for a list of bond periods. Onward distribution to
end stakers is the signer-manager's responsibility.

```ts
import {
  buildCalculateRewards,
  buildClaimRewards,
  burnHeightToRewardCycle,
  currentDistributionCycle,
  distributionCycleToBurnHeight,
  fetchEarned,
  fetchPoxInfo,
} from '@stacks/bitcoin-staking';
import { broadcastTransaction, fetchNonce, signTransaction } from '@stacks/transactions';

const network = 'mainnet';
const signerManager = 'SP000…SIGNER.signer-manager';
const activeBondIndices = [4, 5, 6, 7];
const poxInfo = await fetchPoxInfo({ network });
const distCycle = currentDistributionCycle(poxInfo);
// The STX leg is keyed by the reward cycle that contains
// `calculation-height = distributionCycleToBurnHeight(distCycle) - 1` — the
// burn-block just before the current dist cycle starts. Each reward cycle
// contains two dist cycles, so in closed form: `floor((distCycle - 1) / 2)`.
const settledStxCycle = burnHeightToRewardCycle({
  burnHeight: distributionCycleToBurnHeight({ distributionCycle: distCycle, poxInfo }) - 1,
  poxInfo,
});

// 1. Anyone can settle the cycle once `calculation-height = distribution-cycle-to-burn-height(currentDistCycle) - 1` is reached.
const calcTx = buildCalculateRewards({
  bondIndices: activeBondIndices,
  publicKey: caller.stxPublicKey,
  fee: 10_000n,
  nonce: await fetchNonce({ address: caller.stxAddress, network }),
  network,
});
await broadcastTransaction({ transaction: signTransaction(calcTx, caller.stxPrivateKey), network });

// 2. Signer pulls its share for the just-settled cycle.
//    `get-earned` returns the total newly-earned amount per leg:
//    earned = (shares * (rpt - rptPaid)) / PRECISION + pending.
//
//    `fetchEarned` keys on `{ rewardCycle, bondIndex? }` — omit `bondIndex`
//    for the STX leg, pass it for a bond leg:
//      • STX leg → `fetchEarned({ signerManager, rewardCycle })`
//      • bond leg → `fetchEarned({ signerManager, rewardCycle, bondIndex })`
//    A signer with only one stake type reads one leg. A multi-leg
//    signer-manager (this example) reads both and sums.
//
//    Both legs are keyed by the **reward cycle** matching
//    `calculation-height = distributionCycleToBurnHeight(distCycle) - 1` —
//    i.e. the burn-block just before the start of the current dist cycle.
//    Each reward cycle contains two dist cycles, so the closed-form mapping
//    is `rewardCycle = floor((distCycle - 1) / 2)`.
const stxEarned = await fetchEarned({
  signerManager,
  rewardCycle: settledStxCycle,
  network,
});
const bondEarned = await Promise.all(
  activeBondIndices.map(bondIndex =>
    fetchEarned({ signerManager, rewardCycle: settledStxCycle, bondIndex, network })
  )
);
const totalPending = stxEarned + bondEarned.reduce((acc, e) => acc + e, 0n);
if (totalPending === 0n) return;

const claimTx = buildClaimRewards({
  rewardCycle: settledStxCycle,
  bondIndices: activeBondIndices,
  publicKey: signer.stxPublicKey,
  fee: 10_000n,
  nonce: await fetchNonce({ address: signer.stxAddress, network }),
  network,
});
// claim-rewards returns { stxRewards, bondRewards, bondTotals, totalRewards } —
// per-bond breakdown (each entry: { earned, bondIndex, rewardsPerToken }) is
// available in the print event and the ok-response.
await broadcastTransaction({
  transaction: signTransaction(claimTx, signer.stxPrivateKey),
  network,
});
```

## Lifecycle variants

Bond-agnostic at the top level — a signer-manager typically serves several concurrently-active bonds, each at a different D-day. The `bondIndices` array passed to `calculate-rewards` is the union of all bonds active at `calculation-height` (see `rewards.mdx`: missing one ⇒ `ERR_ACTIVE_BOND_NOT_INCLUDED u33`; out of canonical order ⇒ `ERR_INVALID_BOND_PERIOD_ORDERING u29`).

Per-bond, the leg contributes only between **D0 and D182** of that bond.

| Phase (per bond leg) | What's different |
| --- | --- |
| Pre-D0 | Bond appears in `protocol-bonds` but `is-bond-active-at-height` is false — exclude from `bondIndices` or `calculate-rewards` reverts. `get-earned` returns `0`. |
| D0 → D14 (first distribution) | Bond is active but no settlement has run yet. `fetchEarned` may return `0` or a tiny accrual; `claim-rewards` reverts with `ERR_NO_CLAIMABLE_REWARDS (u32)` until the first `calculate-rewards` lands. |
| D14 → D168 (steady state, ~24 distributions) | The common case. `calculate-rewards` is callable each time `calculation-height = distribution-cycle-to-burn-height(currentDistCycle) - 1` is reached. STX leg + bond leg both accrue. The leg you claim is the matching settled reward cycle — one block before the current dist cycle starts, i.e. `floor((distCycle - 1) / 2)` in closed form. |
| Andon-cord window | Buffer between cycle end and `calculation-height` (the 250-block ops window in upstream tooling; the contract itself has no fixed +250 guard — see `rewards.mdx`). During it, ops can pause via flow 15 — if paused, `calculate-rewards` for that cycle never lands and the leg's earned stays accrued (claimable later, or hard-forked away). |
| D172 → D182 (re-lock window) | Bond is still active for distribution purposes until D182. Continue settling and claiming normally. |
| D183+ | Bond closed; `is-bond-active-at-height` false. Exclude from `bondIndices` going forward. Any historical unclaimed legs remain claimable indefinitely via `claim-rewards` with that bond's last `rewardCycle` — but the bond is no longer settled. |
| All legs zero | `claim-rewards` reverts `ERR_NO_CLAIMABLE_REWARDS (u32)` — gate with the `fetchEarned*` reads. |

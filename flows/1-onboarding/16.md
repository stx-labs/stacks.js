# Flow 16 — Revoke a signer-key grant

Counterpart to flow 6. Removes the on-chain grant binding a signer key
to a signer-manager. Must be sent by the Stacks principal derived from
the signer key — no SIP-018 message needed, the tx-sender check is the
authorization.

```ts
import { buildRevokeSignerGrant } from '@stacks/bitcoin-staking';
import { broadcastTransaction, fetchNonce, signTransaction } from '@stacks/transactions';

const network = 'mainnet';

const tx = buildRevokeSignerGrant({
  signerManager: 'SP000…USER.signer-manager',
  signerKey: signer.publicKey,
  publicKey: signer.stxPublicKey,
  fee: 10_000n,
  nonce: await fetchNonce({ address: signer.stxAddress, network }),
  network,
});

const signed = signTransaction(tx, signer.stxPrivateKey);
await broadcastTransaction({ transaction: signed, network });
```

After this lands, the signer-manager loses its binding to this signer
key. Any `stake` / `register-for-bond` calls routed through that
manager will fail until a fresh `grant-signer-key` is recorded.

## Lifecycle variants

**Bond-agnostic.** Revocation is keyed on `(signerKey, signerManager)` — no bond dimension. But the **effect** ripples across every concurrently-active bond that routes through this manager: when six bonds run in steady state and the manager serves several, all of their future `register-for-bond` calls break the moment this tx lands. Existing positions are *not* unwound (memberships, shares, accrued rewards stay intact); only new flows are gated.

| Phase | What's different |
| --- | --- |
| Pre-revoke | `fetchVerifySignerKeyGrant → true`. Stakers can register through the manager during any open D-7 → D0 window. |
| Revoke tx confirmed | Grant row deleted. Open bond registration windows in any concurrent bond start rejecting via the manager's `validate-stake!`. Already-registered memberships and pending claim flows (7, 15) are unaffected by pox-5 itself — that's a manager-implementation concern: it holds only if the manager's `validate-stake!` doesn't re-check the grant on each call. Rewards continue to settle and can still be claimed under that assumption. |
| After re-grant (flow 6 with a fresh `authId`) | Back to pre-revoke. The previous `authId` cannot be reused (`ERR_SIGNER_KEY_GRANT_USED u12`). |

Errors: `ERR_SIGNER_KEY_GRANT_NOT_FOUND (u17)` if no active grant exists for the pair.

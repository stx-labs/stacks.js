# Flow 20 — Pool operator registers for the bond

From pox-5's perspective there is one staker per pool: the pool
contract itself. It holds the STX + sBTC that members deposited (flow 19) and calls `register-for-bond` as itself via `as-contract`. The
pool's signer-manager is the same contract (or a sibling), already
registered with a signer key (flow 6).

The on-chain SDK call shown here is the _outer_ tx that triggers the
pool contract's own register function. The pool contract internally
re-emits a `contract-call?` to `pox-5::register-for-bond` with
`tx-sender` = pool, `kind: 'sbtc'`, and the aggregated sats total.

```ts
import { fetchBondAllowance } from '@stacks/bitcoin-staking';
import {
  Cl,
  broadcastTransaction,
  fetchNonce,
  makeUnsignedContractCall,
  signTransaction,
} from '@stacks/transactions';

const network = 'mainnet';
const poolContract = 'SP000…COMMUNITY-POOL.v1';

// Sanity: pool's allowance covers the aggregated commitment.
const allowance = await fetchBondAllowance({
  bondIndex: pool.bondIndex,
  address: poolContract,
  network,
});
const totalSats = pool.aggregatedMemberSats;
if (totalSats > allowance) throw new Error('exceeds pool allocation');
```

Notes:

- The pool contract must already be the **registered signer-manager**
  (flow 6) and on the bond's **allowlist** (flow 4). Both are
  prerequisites set off-chain by the Endowment.
- The pool contract is bonded for the full 12-cycle period as a single
  pox-5 membership; member-level accounting lives entirely inside the
  pool.
- For BTC-capacity (native-BTC) bonds there is no pool path in PoX-5 —
  individual whitelisted partners self-enroll via flow 5.

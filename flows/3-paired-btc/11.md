# Flow 11 — Renew a paired bond into the next period

There is no `stake-extend` for paired bonds. Renewal is a fresh
`register-for-bond` for the next bond index, with a new L1 timelock
matching that period's unlock height. The user spends their unlocked
BTC into the new locking address.

```ts
import {
  buildLockingBitcoinAddress,
  buildLockingScript,
  buildRegisterForBond,
  computeBondUnlockHeight,
  fetchPoxInfo,
  firstPox5RewardCycle,
} from '@stacks/bitcoin-staking';
import { broadcastTransaction, fetchNonce, signTransaction } from '@stacks/transactions';

const network = 'mainnet';
const pox = await fetchPoxInfo({ network });
const firstBondPeriodCycle = firstPox5RewardCycle(pox);
if (firstBondPeriodCycle === undefined) throw new Error('pox-5 not yet activated');

// `next` is the bond opening after a BOND_GAP_CYCLES (2-cycle) gap.
// `next.bondIndex` and `next.earlyUnlockSigners` come from fetchBond(...).
const unlockHeight = computeBondUnlockHeight({
  bondIndex: next.bondIndex,
  firstBondPeriodCycle,
  poxInfo: pox,
});
const newLockingAddress = buildLockingBitcoinAddress({
  stxAddress: user.stxAddress,
  unlockHeight,
  publicKey: user.btcPublicKey,
  earlyUnlockBytes: next.earlyUnlockSigners,
  network: 'mainnet',
});
// → user spends their now-unlocked BTC into newLockingAddress

const tx = await buildRegisterForBond({
  bondIndex: next.bondIndex,
  signerManager: 'SP000…USER.signer-manager',
  amountUstx: user.lockedUstx, // can adjust at renewal time
  lockup: {
    kind: 'btc',
    outputs: [{
      height: btcProof.height,
      tx: btcProof.tx,
      outputIndex: 0,
      header: btcProof.header,
      leafHashes: btcProof.leafHashes,
      txCount: btcProof.txCount,
      txIndex: btcProof.txIndex,
      amount: user.lockedSats, // must equal the parsed output value (ERR_INVALID_LOCKUP_AMOUNT u45)
    }],
    unlockBytes: btcProof.unlockBytes,
  },
  publicKey: user.stxPublicKey,
  fee: 10_000n,
  nonce: await fetchNonce({ address: user.stxAddress, network }),
  network,
});

const signed = signTransaction(tx, user.stxPrivateKey);
await broadcastTransaction({ transaction: signed, network });
```

## Atomic relock on the BTC side

Instead of withdrawing then re-locking, spend the prior P2WSH output
directly into `newLockingAddress` once its CLTV has matured. The witness
satisfies the prior unlock-script tail (default tail is `<pubkey>
CHECKSIG`); `nLockTime` must be ≥ the prior unlock height and the input
`nSequence` must be non-final.

```ts
import { Transaction, p2wsh, NETWORK } from '@scure/btc-signer';
import { hex } from '@scure/base';

// Already known from flow 5/9: the script that locked this UTXO. Rebuild
// it via `buildLockingScript({ stxAddress, unlockHeight, unlockBytes,
// earlyUnlockBytes })` with the prior bond's parameters.
const priorScript: Uint8Array = ...;        // from buildLockingScript at lock time
const priorUnlockHeight: number = ...;      // computeBondUnlockHeight for the prior bondIndex
const utxo = (await fetch(`https://mempool.space/api/address/${oldLockingAddress}/utxo`)
  .then(r => r.json()))[0];                  // { txid, vout, value }

const tx = new Transaction({ allowUnknownOutputs: false });
tx.lockTime = priorUnlockHeight;             // satisfies CLTV in priorScript
tx.addInput({
  txid: utxo.txid,
  index: utxo.vout,
  sequence: 0xfffffffe,                      // non-final so nLockTime is enforced (BIP-65)
  witnessUtxo: { script: p2wsh(priorScript).script, amount: BigInt(utxo.value) },
  witnessScript: priorScript,
});
tx.addOutputAddress(newLockingAddress, BigInt(utxo.value) - feeSats, NETWORK);

tx.signIdx(user.btcPrivateKey, 0);           // signs the default <pubkey> CHECKSIG tail
tx.finalize();

const rawHex = hex.encode(tx.extract());
await fetch('https://mempool.space/api/tx', { method: 'POST', body: rawHex });
```

Notes:

- The witness pushes used by `signIdx`/`finalize` work for the default
  `<pubkey> CHECKSIG` tail. A custom unlock-script tail (e.g. an
  early-exit branch) needs explicit `customScripts` / witness assembly.
- Real wallets should fetch a fee estimate; `feeSats` here is a constant.
- This is a single-tx hop on Bitcoin: the output of the spend _is_ the
  new lock.

## Lifecycle variants

**Active window:** D172 → D182 of the *prior* bond (which is D-7 → D0
of the *next* bond — bond periods open every two reward cycles, so the
re-lock window aligns with the next bond's registration window by
design).

| Phase | Behavior |
| --- | --- |
| Prior bond's D-30 → D171 | Too early. Prior bond's L1 UTXO is still locked (CLTV not matured), so the BTC can't be spent into the new address. The L2 `register-for-bond` for `next` would also revert if its own D-7 hasn't opened. |
| D172 → D182 | Re-lock window. Prior L1 CLTV has matured: spend the prior P2WSH directly into the new locking address (atomic-relock variant). Next bond's L2 registration must land before its D0 or it reverts with `ERR_BOND_ALREADY_STARTED`. STX is still locked on L2; renewal extends it without a gap. |
| D183+ | Too late. STX has unlocked, the position is closed. A fresh enrollment (flow 5) is the only path forward — no continuity. |

Cross-flow: this is flow 5 re-issued against the next `bondIndex` plus
a BTC-side hop. The L2 leg shares every error gate with flow 5.

# Flow 10 — Enroll in a paired bond with sBTC (no L1 lock)

Same `register-for-bond` entrypoint as flow 5, but the `lockup`
discriminant flips to `'sbtc'`. The contract calls `lock-sbtc` against
the caller instead of verifying an L1 P2WSH output. No timelock script,
no Bitcoin tx.

Prerequisite: the staker must hold ≥ `intendedSats` of the sBTC FT at
submission. `register-for-bond` calls `lock-sbtc` under `tx-sender`,
which moves sBTC from the staker to the pox-5 contract via
`sbtc-token.transfer`. Acquiring sBTC (peg-in, swap, bridge) is out of
scope for this flow.

```ts
import { buildRegisterForBond, fetchBondAllowance } from '@stacks/bitcoin-staking';
import { broadcastTransaction, fetchNonce, signTransaction } from '@stacks/transactions';

const network = 'mainnet';
const intendedSats = 2_500_000_000n; // 25 BTC equivalent in sBTC

const allowance = await fetchBondAllowance({
  bondIndex: bond.bondIndex,
  address: user.stxAddress,
  network,
});
if (allowance < intendedSats) throw new Error('not allowlisted');

// `amountUstx` must satisfy the contract's min-ustx-for-sats-amount check
// against the bond's stx-value-ratio + min-ustx-ratio; otherwise the call
// reverts with ERR_INSUFFICIENT_STX. Callers typically compute this from
// the bond config via `minUstxForSatsAmount`.
const tx = await buildRegisterForBond({
  bondIndex: bond.bondIndex,
  signerManager: 'SP000…USER.signer-manager',
  amountUstx: bond.requiredUstx,
  lockup: { kind: 'sbtc', sbtcSats: intendedSats },
  publicKey: user.stxPublicKey,
  fee: 10_000n,
  nonce: await fetchNonce({ address: user.stxAddress, network }),
  network,
});
// Callers should attach a post-condition asserting the sBTC FT transfer
// of `intendedSats` from `user.stxAddress` to the pox-5 contract.

const signed = signTransaction(tx, user.stxPrivateKey);
await broadcastTransaction({ transaction: signed, network });
```

## Lifecycle variants

**Active window:** D-7 → D0. Same hard gate as flow 5 (it's the same
`register-for-bond` entrypoint); the lockup discriminant only flips
which validation branch runs inside the contract.

| Phase | Behavior |
| --- | --- |
| D-30 → D-8 | Bond announced, but `register-for-bond` reverts. Allowlist must be in place by D-7. |
| D-7 → D0 | Happy path. The contract calls `lock-sbtc` against `tx-sender`, so the staker's sBTC balance must be ≥ `intendedSats` at submission. No L1 transaction, no SPV proof, no `ERR_INVALID_LOCKUP_AMOUNT` path. Capacity (`ERR_TOO_MUCH_SATS`) and allowlist (`ERR_NOT_ALLOWLISTED`) still apply. |
| D0+ | `ERR_BOND_ALREADY_STARTED (u43)`. No rollover. |

Once registered, partial withdrawal during the bond is possible via the
sBTC-only `unstake-sbtc(signer-manager, amount-to-withdrawal-sats)`
(`ERR_CANNOT_UNSTAKE_SBTC (u38)` for L1-locked members) — see
`buildUnstakeSbtc` in `build.ts`. That flow is sBTC-specific and has no
analogue for paired native-BTC bonds.

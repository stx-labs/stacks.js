# Flow 5 — Enroll in a paired-BTC bond

The user pairs BTC with STX in a protocol bond. Three steps: confirm the
caller's allowlist entry covers their intended sats (seeded by the
Endowment in flow 4); construct the L1 timelock address and send BTC to
it; then register on L2.

Note: `register-for-bond` reverts with `ERR_ALREADY_STAKED (u19)` if the
caller currently has an active solo-STX stake (`pox-5.clar` line ~558) —
solo-STX and paired-BTC positions are mutually exclusive per principal.

```ts
import {
  bondPeriodToRewardCycle,
  buildLockingBitcoinAddress,
  buildRegisterForBond,
  computeBondUnlockHeight,
  fetchBondAllowance,
  fetchPoxInfo,
  firstPox5RewardCycle,
  minUstxForSatsAmount,
} from '@stacks/bitcoin-staking';
import { broadcastTransaction, fetchNonce, signTransaction } from '@stacks/transactions';

const network = 'mainnet';
const pox = await fetchPoxInfo({ network });
const firstBondPeriodCycle = firstPox5RewardCycle(pox);
if (firstBondPeriodCycle === undefined) throw new Error('pox-5 not yet activated');
const intendedSats = 2_500_000_000n;  // 25 BTC

// 1. Allowlist check — fails fast if the user isn't approved for this bond
//    or if their cap is below what they want to lock.
const allowance = await fetchBondAllowance({
  bondIndex: bond.bondIndex,
  address: user.stxAddress,
  network,
});
if (allowance < intendedSats) {
  throw new Error(`not allowlisted for ${intendedSats} sats on bond ${bond.bondIndex} (cap: ${allowance})`);
}

// 2. L1: derive the timelock address. The user funds it from their BTC wallet.
const unlockHeight = computeBondUnlockHeight({
  bondIndex: bond.bondIndex,
  firstBondPeriodCycle,
  poxInfo: pox,
});
const lockingAddress = buildLockingBitcoinAddress({
  stxAddress: user.stxAddress,
  unlockHeight,
  publicKey: user.btcPublicKey,            // tail: <pubkey> OP_CHECKSIG
  earlyUnlockBytes: bond.earlyUnlockSigners, // from fetchBond(...)
  network: 'mainnet',
});
// → user sends e.g. 25 BTC to lockingAddress out-of-band

// 3. L2: register for the bond. The `outputs` list is the SPV proof
//    shape: per-output { height, tx, outputIndex, header, leafHashes,
//    txCount, txIndex, amount }. `amount` must equal the parsed output
//    value or the contract reverts with ERR_INVALID_LOCKUP_AMOUNT (u45).
const tx = await buildRegisterForBond({
  bondIndex: bond.bondIndex,
  signerManager: 'SP000…USER.signer-manager',
  amountUstx: minUstxForSatsAmount({
    sats: intendedSats,
    stxValueRatio: bond.stxValueRatio,
    minUstxRatioBps: bond.minUstxRatioBps,
  }),
  lockup: {
    kind: 'btc',
    outputs: [{
      height: btcProof.height,
      tx: btcProof.tx,
      outputIndex: 0,
      header: btcProof.header,
      leafHashes: btcProof.leafHashes,
      txCount: btcProof.txCount,
      txIndex: btcProof.txIndex,
      amount: intendedSats,
    }],
    unlockBytes: btcProof.unlockBytes, // tail pushed in both branches
  },
  publicKey: user.stxPublicKey,
  fee: 10_000n,
  nonce: await fetchNonce({ address: user.stxAddress, network }),
  network,
});

const signed = signTransaction(tx, user.stxPrivateKey);
await broadcastTransaction({ transaction: signed, network });
```

## Lifecycle variants

**Active window:** D-7 → D0 (the registration window). `register-for-bond`
asserts `burn-block-height < bond-start-height`; outside this window the
L2 tx reverts. Two further on-chain prerequisites bracket the call:

| Phase | Behavior |
| --- | --- |
| D-30 → D-8 | Bond is `announced` but registration is not open yet. The L1 deposit can already be sent (the script is deterministic from `bondIndex` + `earlyUnlockSigners`), but step 3 will revert until D-7. |
| D-7 → D0 | Happy path. Allowlist must cover `intendedSats`; the L1 UTXO must be confirmed before the L2 tx is mined or step 3 reverts with `ERR_INVALID_LOCKUP_AMOUNT (u45)` (declared `amount` ≠ parsed output value) / `ERR_INVALID_LOCKUP_SCRIPT (u42)`. Bond capacity may also be exhausted (`ERR_TOO_MUCH_SATS`). |
| D0+ | Bond has started. Step 3 reverts with `ERR_BOND_ALREADY_STARTED (u43)`. No grace period — the BTC stays locked at the script's `unlockHeight` (D172). No auto-rollover; the user must wait until that bond's re-lock window to redeploy. |

Prereq: flow 4 (Endowment-side allowlist seeding) must have written the
caller into this bond's allowlist by D-7. |

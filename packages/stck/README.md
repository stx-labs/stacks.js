# @stacks/stck

Typed contract call helpers for Stacks smart contracts. Wraps `makeUnsignedContractCall` and `fetchCallReadOnlyFunction` from `@stacks/transactions` with compile-time type safety using TypeScript types generated by Clarinet's `typegen` command.

This package ships two parallel POC implementations — one for each style of type generation supported by Clarinet — so teams can evaluate the developer experience side by side.

## Install

```bash
npm install @stacks/stck @stacks/transactions
```

You also need Clarinet's `typegen` to generate type files from your `Clarinet.toml`:

```bash
clarinet typegen
# Produces:
#   generated/typed/  (Approach A — bundled value+type per contract)
#   generated/abi/    (Approach B — ABI as const + utility types)
```

## The bundle (Approach A)

Clarinet emits one file per contract that exports a single `<contract>Contract` object — the runtime ABI literal *fused* with a phantom symbol-keyed brand pointing at the typed contract interface. One import, no explicit generic at the call site.

```typescript
// generated/typed/counter.ts (auto-generated)

export type AddArgs = [UIntCV];
export type AddReturn = ResponseOkCV<BooleanCV> | ResponseErrorCV<UIntCV>;
export type GetCountArgs = [];
export type GetCountReturn = UIntCV;

export interface CounterContract {
  functions: {
    add: { args: AddArgs; return: AddReturn };
    getCount: { args: GetCountArgs; return: GetCountReturn };
    // ...
  };
  // ...
}

// Bundled value: as-const ABI literal + phantom brand
export const counterContract = { /* full ABI */ } as const as
  typeof _base & { readonly [__contractBrand]: CounterContract };
```

Naming conventions:
- **Approach A (camelCase keys):** function keys on `CounterContract` are camelCased (`get-count` → `getCount`).
- **Approach B (kebab keys):** the ABI literal preserves the original Clarity names (`"get-count"`).

Arguments are **positional tuples** in the order declared by the Clarity function — e.g. `[5]` for a single-arg call, `[]` for no-arg.

## Usage

### Approach A: `makeUnsignedContractCallA`

```typescript
import { makeUnsignedContractCallA } from "@stacks/stck";
import { Cl } from "@stacks/transactions";
import { counterContract } from "./generated/typed/counter";

const tx = await makeUnsignedContractCallA(counterContract, {
  contract: "ST1PQHQ....counter",
  functionName: "increment",         // autocomplete: "increment" | "add" | ...
  functionArgs: [Cl.uint(5)],         // typed: [UIntCV]
  publicKey: "...",
});
```

A single argument (`counterContract`) carries both the typed interface and the runtime ABI. No explicit generic needed.

### Approach B: `makeUnsignedContractCallB`

```typescript
import { makeUnsignedContractCallB } from "@stacks/stck";
import { Cl } from "@stacks/transactions";
import { counterContract } from "./generated/typed/counter";

const tx = await makeUnsignedContractCallB(counterContract, {
  contract: "ST1PQHQ....counter",
  functionName: "increment",         // autocomplete: "increment" | "get-count"
  functionArgs: [Cl.uint(5)],
  publicKey: "...",
});
```

Same shape; the type inference path differs. B reads everything from the `as const` ABI literal; A reads it from the brand. Both consume the same bundle.

## Comparison

|                              | Approach A                                       | Approach B                                                  |
|------------------------------|--------------------------------------------------|-------------------------------------------------------------|
| **Function name format**     | camelCase (`"getCount"`)                         | original Clarity (`"get-count"`)                            |
| **Argument form**            | positional tuple (`[5]`)                         | positional tuple (`[5]`)                                    |
| **Bundle required?**         | Yes — single import                              | Yes — same import (B doesn't use the brand)                 |
| **Generics**                 | Inferred from the bundle's brand                 | Inferred from the `as const` ABI                            |
| **Error messages**           | Clear, named aliases (`AddArgs`)                 | Compact structural — no ABI dump                            |
| **Cmd-click to definition**  | Yes — types resolve to named aliases             | Type expansion shown in hover, not navigable                |
| **TS compiler overhead**     | None (plain type aliases)                        | Some (conditional type evaluation)                          |
| **Type drift risk**          | Theoretical (mitigated by tests)                 | Impossible by construction                                  |

### When to choose which

**Approach A** is the pragmatic choice if you prize:
- Clear, readable error messages at the point of failure
- The ability to cmd-click into a type to understand what's expected
- Self-documenting generated files that are easy to review in PRs

**Approach B** is compelling if you prize:
- The ABI as a single source of truth (types can't drift)
- Reusability — the utility types work on any ABI, including ones fetched from `v2/contracts/interface/:address/:name` at runtime
- Minimal generated artifacts (just data, no derived types)

For most teams shipping Clarity contracts, Approach A's error clarity is the decisive factor.

## Contract wrapper

Both approaches also ship a higher-level **contract wrapper**: bind `contract`, `publicKey`, and (optionally) `network` + `client` once, then call functions by name with JS primitives (auto-converted to ClarityValues using the ABI). Two methods today — `makeUnsignedContractCall` and `fetchCallReadOnlyFunction` — and the factory is shaped so that `makeSignedContractCall`, deploy helpers, etc. can be added later under the same surface.

`makeUnsignedContractCall` only accepts **public** functions and `fetchCallReadOnlyFunction` only accepts **read-only** functions — the type level enforces both, so the wrong call style is a compile-time error.

### Approach A — `contractA(bundle, bindings)`

```typescript
import { contractA } from "@stacks/stck";
import { counterContract } from "./generated/typed/counter";

const counter = contractA(counterContract, {
  contract: "ST1PQHQ....counter",
  publicKey: "...",
  network: "testnet",                  // optional — defaults to "mainnet"
  client: { baseUrl: "https://..." },  // optional — overrides the network default
});

// JS primitives are coerced via the ABI:
await counter.makeUnsignedContractCall("add", [5]);

// Pre-built ClarityValues are accepted as an escape hatch:
await counter.makeUnsignedContractCall("add", [Cl.uint(5)]);

// Optional trailing opts override defaults or supply fee / nonce / postConditions:
await counter.makeUnsignedContractCall("add", [5], { fee: 1000n, nonce: 7n });

// Read-only calls go through the network — return type narrows to the precise CV:
const count = await counter.fetchCallReadOnlyFunction("getCount", []);
//    ^? UIntCV

// `senderAddress` defaults to the address derived from the bound `publicKey`. Override per call:
await counter.fetchCallReadOnlyFunction("getCount", [], { senderAddress: "ST2..." });
```

Method names use **camelCase** (`getCount`), matching the bundled contract interface.

### Approach B — `contractB(bundle, bindings)`

```typescript
import { contractB } from "@stacks/stck";
import { counterContract } from "./generated/typed/counter";

const counter = contractB(counterContract, {
  contract: "ST1PQHQ....counter",
  publicKey: "...",
  network: "testnet",
  client: { baseUrl: "https://..." },
});

await counter.makeUnsignedContractCall("add", [5]);
await counter.makeUnsignedContractCall("add", [Cl.uint(5)]);
await counter.makeUnsignedContractCall("add", [5], { fee: 1000n });

// Read-only — kebab-case names match the ABI:
const count = await counter.fetchCallReadOnlyFunction("get-count", []);
//    ^? UIntCV
await counter.fetchCallReadOnlyFunction("get-count", [], { senderAddress: "ST2..." });
```

Method names use **original Clarity kebab-case** (`"get-count"`), matching the ABI literal.

### JS-primitive → ClarityValue coercion

Both wrappers coerce each named arg through `toClarityValue(abiType, value)` and emit positional `ClarityValue[]` in the ABI's declared order. Coercion table:

| ABI type                  | Accepted JS input                  | Produced CV         |
|---------------------------|------------------------------------|---------------------|
| `uint128` / `int128`      | `number \| bigint`                 | `Cl.uint` / `Cl.int`|
| `bool`                    | `boolean`                          | `Cl.bool`           |
| `principal`               | `string`                           | `Cl.principal`      |
| `none`                    | `null`                             | `Cl.none`           |
| `buffer`                  | `Uint8Array \| string` (hex)       | `Cl.buffer` / `Cl.bufferFromHex` |
| `string-utf8`             | `string`                           | `Cl.stringUtf8`     |
| `string-ascii`            | `string`                           | `Cl.stringAscii`    |
| `optional<T>`             | `T \| null`                        | `Cl.some` / `Cl.none` |
| `list<T>`                 | `T[]`                              | `Cl.list`           |
| `tuple<{k: T, ...}>`      | `{ k: T, ... }`                    | `Cl.tuple`          |
| (any)                     | pre-built `ClarityValue`           | passthrough         |

## Utility types (Approach B)

The package owns and exports the type-level machinery used by Approach B. These work on any `as const` ABI matching the Clarity contract interface JSON shape — not just Clarinet-generated ones.

```typescript
import type { Args, Return, FunctionNames, AbiTypeToCv } from "@stacks/stck";
import { counterContract } from "./generated/typed/counter";

type Fns = FunctionNames<typeof counterContract>;
//   ↑ "increment" | "add" | "get-count" | ...

type IncArgs = Args<typeof counterContract, "increment">;
//   ↑ [UIntCV]

type IncRet = Return<typeof counterContract, "increment">;
//   ↑ ResponseOkCV<BooleanCV> | ResponseErrorCV<UIntCV>

type CV = AbiTypeToCv<{ optional: "uint128" }>;
//   ↑ NoneCV | SomeCV<UIntCV>
```

## API

```typescript
// Bundle inference
type ContractBundle;                   // structural shape of any generated bundle
type ExtractContractInterface<B>;      // pulls the brand interface out of a bundle

// Approach A
function makeUnsignedContractCallA<B extends ContractBundle, F extends PublicFunctionNames<B>>(
  bundle: B,
  options: TypedOptionsA<B, F>
): Promise<StacksTransactionWire>;

// Approach B
function makeUnsignedContractCallB<ABI, F extends FunctionNames<ABI, "public">>(
  abi: ABI,
  options: TypedOptionsB<ABI, F>
): Promise<StacksTransactionWire>;

// Approach B utility types
type AbiTypeToCv<T>;
type Args<ABI, FnName>;                // positional tuple of CVs
type Return<ABI, FnName>;
type FunctionNames<ABI, Access?>;

// Contract wrapper
function contractA<B extends ContractBundle>(
  bundle: B,
  bindings: {
    contract: `${string}.${string}`;
    publicKey: string;
    network?: StacksNetworkName | StacksNetwork;  // default "mainnet"
    client?: ClientOpts;                          // { baseUrl?, fetch? }
  }
): ContractClientForBundleA<B>;

function contractB<const ABI>(
  abi: ABI,
  bindings: {
    contract: `${string}.${string}`;
    publicKey: string;
    network?: StacksNetworkName | StacksNetwork;
    client?: ClientOpts;
  }
): ContractClientB<ABI>;

// Per-call options for read-only calls
type PerCallReadOnlyOptions = {
  /** Override the auto-derived sender (defaults to the address of the bound publicKey). */
  senderAddress?: string;
};

// JS-primitive → ClarityValue coercion (driven by the ABI atom type)
function toClarityValue(abiType: AbiAtomType, value: unknown): ClarityValue;
function coerceArgs(
  abiArgs: { type: AbiAtomType }[],
  values: readonly unknown[]
): ClarityValue[];
function isClarityValue(v: unknown): v is ClarityValue;
```

## Status

This package contains two POC implementations for evaluation. One approach is expected to be selected as the canonical implementation before stabilization.
